# Business Entity Model

Enterprise business object model for the platform: which entities exist or are planned, what each is responsible for, how they relate, and what's actually implemented today versus proposed. Product-approved as of this writing (Business Entity Model final revision). See `CLAUDE_CONTEXT.md`'s "Product Authorities" section for how this document relates to `ProductVision.md`/`ProductBlueprint.md`, and `PROJECT_ARCHITECTURE.md`'s "Relationship to Product Governance" section for how it relates to the architecture docs.

This document is documentation only — it does not itself implement any collection, rule, or schema described below. Sections 10-12 call out explicitly what is and isn't built yet.

## 4. Reorder Request

Inventory Management's request to replenish a Part, originated from the Inventory workspace's Needs Reorder queue (Sprint 2.1.2). Dual-classified in `PROJECT_ARCHITECTURE.md`'s Enterprise Platform Classification Model as both a **Business Object** (this entry) and an **Operational Workflow Object** — the same non-exclusive pairing already established for Work Order (Section 3 / Section 8's relationship diagram). Its creation is also this platform's first new named **Platform Event** ("Reorder Requested") since that classification model was adopted — the record's `createdAt` timestamp *is* that event's immutable fact, not a separate log entry.

Fields (`reorder_requests` collection): `partId`, `urgency`, `recommendedQty`, `status`, `requestedBy`, `createdAt` (Sprint 2.1.3); `reviewedBy`, `reviewedAt`, `reviewDecision`, `reviewNotes` (Sprint 2.1.4 — reserved as `null` at creation, populated once a Reorder Request is reviewed); `currentOwner` (Sprint 2.1.5); `assignedToUserId`, `assignedBy`, `assignedAt` (Sprint 2.1.6 — reserved as `null` at creation, populated once a Reorder Request is assigned); `purchasingStartedAt`, `purchasingStartedBy` (Sprint 2.1.7 — reserved as `null` at creation, populated once purchasing starts); `purchasingNotes`, `vendorContacted`, `expectedAvailabilityDate`, `lastPurchasingUpdateAt`, `lastPurchasingUpdateBy` (Sprint 2.1.8 — reserved as `null` at creation, populated/overwritten each time a purchasing progress update is posted); `purchaseOrderId`, `orderedBy`, `orderedAt` (Sprint 2.1.10 — reserved as `null` at creation, populated once a Purchase Order is recorded). `status` and `reviewDecision` are deliberately separate fields: `status` is this object's general lifecycle position, which keeps moving after review (see below); `reviewDecision` is a permanent historical fact of what was decided during review, never overwritten once set. `currentOwner` is a coarse, role-level ownership marker; `assignedToUserId` (Sprint 2.1.6) is the platform's first **per-user** workflow ownership field. `purchasingStartedBy` (Sprint 2.1.7), `lastPurchasingUpdateBy` (Sprint 2.1.8), and `orderedBy` (Sprint 2.1.10) all record who acted — expected to equal `assignedToUserId` (server-enforced, see below) rather than adding a second, redundant identity concept. Unlike every other field on this object, `purchasingNotes`/`vendorContacted`/`expectedAvailabilityDate`/`lastPurchasingUpdateAt`/`lastPurchasingUpdateBy` are **overwritten** on each update, not append-only — this is a "latest known state" progress snapshot, not a workflow history log entry; no history of prior updates is kept. `purchaseOrderId` always equals this document's own ID (a Reorder Purchase Order's document ID is its linked Reorder Request's ID — see Section 4b) -- it is a link, not a separate identifier scheme.

Status values: `PENDING_REVIEW` (Sprint 2.1.3) → `READY_FOR_PARTS_MANAGER` on approval, or `REJECTED` (Sprint 2.1.5, terminal — an already-decided request cannot be re-reviewed) → `ASSIGNED_TO_PARTS_ASSOCIATE` (Sprint 2.1.6) → `PURCHASING_IN_PROGRESS` (Sprint 2.1.7 -- a request can receive any number of progress updates while it stays in this status) → `ORDERED` (Sprint 2.1.10, terminal for this sprint -- no transition out of it yet). `reviewDecision` still records `APPROVED`/`REJECTED` as the permanent decision even after `status` moves well past `READY_FOR_PARTS_MANAGER`. `reviewNotes` is required when `reviewDecision` is `REJECTED`, optional when `APPROVED`. `currentOwner` values: `INVENTORY` (from creation through review) → `PARTS_MANAGER` (on approval) → `PARTS_ASSOCIATE` (on assignment, and unchanged by starting purchasing, posting progress updates, or recording a Purchase Order — same person's work throughout, not a hand-off; rejection leaves ownership with Inventory). `assignedToUserId`/`assignedBy`/`assignedAt` are set together, only on the `READY_FOR_PARTS_MANAGER` → `ASSIGNED_TO_PARTS_ASSOCIATE` transition; `assignedBy` is the Parts Manager's own uid (server-enforced), `assignedToUserId` is a manually-entered target uid (no client-side user list to pick from — `firestore.rules`' `users/{userId}` read is self-only, mirrors PT-001's `assignTechnicianToUser.js` pattern). `purchasingStartedAt`/`purchasingStartedBy` are set together, only on the `ASSIGNED_TO_PARTS_ASSOCIATE` → `PURCHASING_IN_PROGRESS` transition, and only by the request's own `assignedToUserId`. `purchasingNotes`/`vendorContacted`/`expectedAvailabilityDate`/`lastPurchasingUpdateAt`/`lastPurchasingUpdateBy` (Sprint 2.1.8) are set together on a `PURCHASING_IN_PROGRESS` → `PURCHASING_IN_PROGRESS` update (status unchanged — the first non-transition write on this object), also only by the request's own `assignedToUserId`. `purchaseOrderId`/`orderedBy`/`orderedAt` (Sprint 2.1.10) are set together, only on the `PURCHASING_IN_PROGRESS` → `ORDERED` transition, atomically alongside creating the linked Reorder Purchase Order record (Section 4b) in a single Firestore transaction — also only by the request's own `assignedToUserId`, enforced in `firestore.rules` (`request.auth.uid == resource.data.assignedToUserId`), not just application code.

Ownership: **Inventory Management remains the sole owner of the Reorder Request's Business Capability** — Sprints 2.1.5 through 2.1.8 transfer/advance successive stages of *workflow* ownership and progress within Inventory; Sprint 2.1.10's `ORDERED` transition hands the purchasing execution record to **Procurement** as the owning Business Capability for the new Reorder Purchase Order object (Section 4b), while Reorder Request itself remains Inventory's. Reviewing, assigning, starting purchasing, posting progress updates, and recording a Purchase Order are all still the existing admin/dispatcher role, not a new permission tier; there is no separate Parts Manager or Parts Associate auth role — `currentOwner`/`assignedToUserId` are data-level markers the Parts Manager Queue, Parts Associate Queue, and Notification Panel filter on. Starting purchasing, posting updates, and recording a Purchase Order are all further restricted to one specific person within that role, not just gated by role. Sprint 2.1.10 deliberately does not implement Supplier/Vendor Management, a vendor catalog, or a pricing engine — `supplierName` on the Reorder Purchase Order remains manually entered free text.

Write path: client-direct-write via `domain/inventoryReorderRequests.js` (mirrors the `accounts`/`locations`/`contacts` pattern) — `createReorderRequest()` (Sprint 2.1.3), `reviewReorderRequest()` (Sprint 2.1.4, extended Sprint 2.1.5), `assignReorderRequest()` (Sprint 2.1.6), `startPurchasing()` (Sprint 2.1.7), and `updatePurchasingProgress()` (Sprint 2.1.8) are its writers for the first four transitions. The fifth transition (`PURCHASING_IN_PROGRESS` → `ORDERED`) is written by a different file, `domain/reorderPurchaseOrders.js`'s `recordPurchaseOrder()`, since it must atomically also create the linked Reorder Purchase Order record — see Section 4b for the full atomicity design. No Cloud Function anywhere in this chain. `firestore.rules`' update rule enforces five paths in sequence (`PENDING_REVIEW` → `READY_FOR_PARTS_MANAGER`/`REJECTED`; `READY_FOR_PARTS_MANAGER` → `ASSIGNED_TO_PARTS_ASSOCIATE`; `ASSIGNED_TO_PARTS_ASSOCIATE` → `PURCHASING_IN_PROGRESS`, assignee-only; `PURCHASING_IN_PROGRESS` → `PURCHASING_IN_PROGRESS`, also assignee-only and a non-transition path; `PURCHASING_IN_PROGRESS` → `ORDERED`, assignee-only), each with its own field-matching/`affectedKeys()` constraints, plus the immutability of the original request fields and of every earlier stage's fields once a later write happens.

Status: Core (V2), Sprint 2.1.3-2.1.10.

## 4a. Inventory Action

A human-logged note about a stock adjustment against a Part -- Receive Stock, Adjust Stock, or Correct Mistake -- introduced by Sprint 2.1.9. Deliberately a **separate, new collection** (`inventory_actions`), not a modification of `inventory_transactions` (Epic 2D/3, ADR-003): that ledger remains Admin-SDK-only, Work-Order-driven, and untouched. This mirrors how Reorder Request (Section 4) was added as a new parallel Business Object rather than reshaping an existing one — "no second inventory system competing with the ledger" (`CLAUDE_CONTEXT.md` rule 6) means not reusing or mutating `inventory_transactions`, not that no other inventory-adjacent collection may ever exist.

**Logged-only, as of Sprint 2.1.9 -- does NOT change stock.** An Inventory Action is an audit note, not a live inventory adjustment. `computeAvailableStockByPart()`/`inventoryAnalyticsEngine.ts` are entirely untouched by this object and remain sourced solely from `inventory_transactions`. The UI (`PartDetail.jsx`'s "Inventory Action Log" card) says this explicitly ("This records an audit note only. It does not update stock yet.") rather than implying an operational Receive Stock/Adjust Stock/Correct Mistake actually moved inventory — a first version of this card did imply that, caught in PR #76's architectural review before merge.

**Backlog note (not yet scoped): apply Inventory Actions to `inventory_transactions` through a Cloud-Function-mediated trusted write path once Firebase Blaze is enabled.** Direct client writes to `inventory_transactions` must never be added (it stays Admin-SDK-only, per ADR-003) — a trusted server-side write (mirroring `functions/src/inventoryService.ts`'s `reserveParts`/`releaseParts`/`consumeParts`) is the only sanctioned way to make these actions actually move stock. This is blocked on the platform's standing decision not to enable Blaze yet (`CLAUDE_CONTEXT.md`), not on anything Sprint 2.1.9 could have built differently — building an undeployed Cloud Function now would only add unverifiable code, same limbo `createWorkOrder`/`transitionWorkOrder` already sit in.

Fields (`inventory_actions` collection): `partId`, `transactionType`, `quantityDelta`, `reason`, `notes`, `createdBy`, `createdAt`. `transactionType` values: `RECEIVE_STOCK` (requires `quantityDelta > 0`), `ADJUST_STOCK` (any non-zero `quantityDelta`, positive or negative), `CORRECT_MISTAKE` (any non-zero `quantityDelta`, but requires both `reason` and `notes` to be non-empty). All three constraints are enforced in `domain/inventoryActions.js`'s `recordInventoryAction()` -- the sole write path -- not just in the UI. `quantityDelta` here is what the user reports happened, for the record -- it is not (yet) reflected in any computed stock number.

Unlike Reorder Request, this object has **no status field and no workflow** -- each action is a single, complete, append-only record (a log entry, not a multi-step process). There is no update or delete path: correcting a mistake means recording another action, never editing history.

Ownership: same as Reorder Request -- Inventory Management owns the capability; logging an action is gated to the existing admin/dispatcher role, no new permission tier.

Write path: client-direct-write via `domain/inventoryActions.js` (mirrors `domain/inventoryReorderRequests.js`'s pattern) -- `recordInventoryAction()` is its only export that writes. No Cloud Function. `firestore.rules`' `inventory_actions` rule allows `create` (admin/dispatcher only, no field-level validation -- same posture as `reorder_requests`' create rule) and denies `update`/`delete` unconditionally.

Read path: `hooks/useInventoryActions.js`'s `useInventoryActionsForPart(partId)` -- realtime (`onSnapshot`, server-side `where("partId", ...)` filtered) from the start, unlike Reorder Request's hooks which needed a post-2.1.7 bug fix to become realtime. Shown on `PartDetail.jsx`'s Inventory Action Log card (most recent 10, sorted client-side by `createdAt` descending), with a persistent "not applied to stock" note.

Status: Core (V2), Sprint 2.1.9 -- logged-only; not yet applied to the inventory ledger.

## 4b. Reorder Purchase Order

A minimal purchasing execution record created by the assigned Parts Associate to record that a Reorder Request has actually been ordered from a supplier -- introduced by Sprint 2.1.10. Classified as both a **Business Object** and an **Operational Workflow Object** (same non-exclusive pairing as Work Order and Reorder Request); its creation is also a new named **Platform Event** ("Order Recorded"). Procurement is the owning Business Capability for this object; Reorder Request remains the originating demand record, and this object is its purchasing execution record.

**Deliberately NOT the existing `purchase_orders` collection** (Epic 5, Procurement + Supplier Management -- Section 3's object model table, Admin-SDK-only, Supplier-linked, related to Part via line items). That collection is a full future Purchase Order entity this platform hasn't built execution for yet; reusing its name or path would either collide with it (it denies all client writes unconditionally) or misleadingly imply this minimal, single-Reorder-Request-linked record is that entity. This object is named **"Reorder Purchase Order"** specifically to keep the two unambiguous, and lives in a new, separate collection: `reorder_purchase_orders`.

**Atomicity**: creating a Reorder Purchase Order and transitioning its linked Reorder Request to `ORDERED` happen in a single Firestore client-side transaction (`runTransaction()`) -- both writes commit together or neither does. This did not require a Cloud Function or Admin SDK: unlike Sprint 2.1.9's `inventory_transactions` blocker (which needed a trusted server write because that collection is Admin-SDK-only), both collections involved here (`reorder_requests`, `reorder_purchase_orders`) are already client-direct-write-with-rules collections, so a standard Firestore Web SDK transaction is sufficient and fully rules-enforceable.

**Duplicate prevention**: each Reorder Purchase Order's document ID *is* its linked Reorder Request's ID (not an auto-generated ID). This lets `firestore.rules` enforce "no duplicate Purchase Order per Reorder Request" through Firestore's own create-vs-update distinction -- a second write to the same ID is evaluated as an `update`, which the rule denies unconditionally -- rather than relying only on an application-level check that a race could defeat.

Fields (`reorder_purchase_orders` collection): `reorderRequestId`, `partId`, `supplierName`, `externalPoNumber`, `orderedQuantity`, `orderedDate`, `expectedArrivalDate` (optional), `status` (always `ORDERED` -- this object has no further workflow this sprint), `createdBy`, `createdAt`. `supplierName` is manually entered free text -- no Supplier/Vendor Management object, no vendor catalog, no pricing engine exist yet; this is intentionally the same "manual text field, not a picker" posture as `assignedToUserId` (PT-001's `assignTechnicianToUser.js` pattern) and matches this sprint's explicit scope.

Reorder Request relationship: `reorder_requests` gains `purchaseOrderId` (always equal to the linked Reorder Purchase Order's ID, i.e. its own ID), `orderedBy` (the assignee's own uid, server-enforced), and `orderedAt`, set together only on the `PURCHASING_IN_PROGRESS` → `ORDERED` transition -- the platform's fifth transition on this object, following the same assignee-only, prior-fields-pinned-unchanged discipline as Sprints 2.1.7/2.1.8's transitions. All review, assignment, purchasing-start, and purchasing-update fields are preserved unchanged by this transition.

Ownership: same as Reorder Request -- gated to the existing admin/dispatcher role, further restricted to the specific assigned Parts Associate for the actual write (same per-user restriction pattern as `startPurchasing()`/`updatePurchasingProgress()`). No new permission tier.

Write path: client-direct-write via `domain/reorderPurchaseOrders.js`'s `recordPurchaseOrder()` -- the sole writer of this collection and the sole place `reorder_requests` ever transitions to `ORDERED`. No Cloud Function. `firestore.rules` gains a new `reorder_purchase_orders` match block (admin/dispatcher `create` only, validated against the linked Reorder Request's current state via `get()`, `update`/`delete` denied unconditionally) and a fifth `reorder_requests` update path (`PURCHASING_IN_PROGRESS` → `ORDERED`, assignee-only, all earlier-stage fields pinned immutable).

Read path: `hooks/useReorderPurchaseOrders.js`'s `usePurchaseOrderForReorderRequest(reorderRequestId)` -- realtime (`onSnapshot` on the single document, since its ID is already known) from the start. Shown on `PartDetail.jsx`'s "Reorder Request -- Ordered" card once a request reaches `ORDERED`; the "Record Purchase Order" form is shown (assignee-only) alongside the existing purchasing-update card while `PURCHASING_IN_PROGRESS`. Removal from the Parts Associate "In Progress" queue happens automatically through the existing realtime status filtering -- no new queue-removal logic needed.

Status: Core (V2), Sprint 2.1.10.

## 5. Company

Represents the operating business *using* the platform — distinct from a customer Account, which represents who the operating business serves.

**Explicitly separate from Account.** An Account is who the operating company does work for; Company is who's running the platform. Today, with a single operating company, this distinction is latent — Company becomes load-bearing only if the platform ever serves multiple operating companies from one deployment, per `ProductVision.md`'s long-term multi-tenant scope.

Future relationships:

```
Company
    ├── Employees
    ├── Warehouses
    ├── Vehicles
    ├── Regions
    ├── Financials
    └── Company Settings
```

Status: Future roadmap entity.

## 6. Vehicle

Represents a mobile inventory storage location, typically carried by field personnel. Intentionally broader than a single vehicle type — can represent:

- Service Truck
- Van
- Car
- Trailer
- Mobile Warehouse
- Future asset types not yet anticipated

Same relationship shape as Warehouse (many-to-many with Parts). No prior implementation exists to migrate — the demo/in-memory "truck" concept in `Inventory.jsx` is unrelated and out of scope for this entity.

Status: Future roadmap entity.

## 7. Service Contract

Governs the ongoing service relationship terms for an Account — service levels, maintenance cadence, warranty coverage, and billing rules — as a distinct entity from any single Work Order.

Relationship chain:

```
Account
    ↓
Service Contract
    ↓
PM Schedule
    ↓
Work Orders
```

A Service Contract belongs to one Account; it generates a PM Schedule (recurring maintenance cadence); the PM Schedule is what produces/triggers Work Orders over time — Work Orders remain the actual unit of field service execution, unchanged.

Future responsibilities (documented for coherence, not designed in detail):
- SLA (response-time commitments)
- Preventive Maintenance (cadence, scope)
- Warranty (coverage terms, expiration)
- Billing Rules (rate schedules, contract pricing vs. time-and-materials)
- Coverage (inclusions/exclusions)
- Response Times
- Included Labor
- Included Parts

**Status: Future roadmap entity, Version 3+.** Documented now so the Account→Work Order relationship (Core, Version 2) is understood as the simple case of a longer chain the platform will eventually support, without near-term work needing to anticipate it.

## 8a. Employee

**Approved architecture, not yet implemented.** This section formalizes the Employee/User/Firebase-Authentication identity split governed by `PROJECT_ARCHITECTURE.md`'s "Person Assignment Platform Service Standard" (Enterprise Platform Classification Model, Section A). Nothing in this section describes deployed code, a live collection, or a built UI — see Section 9 for current build status.

- **Employee** (proposed collection: `employees/{employeeId}`) is the authoritative workforce/business identity. An Employee may exist without any linked application access.
- **User** (`users/{uid}`, existing, already real) is the application-access identity — who can sign in and what security role they hold. Linked to an Employee via `employeeId` where applicable; disabling a User's access must never delete the linked Employee or its historical assignment records.
- **Firebase Authentication** is the credential/session authority only — it is *not* a Business Object and gets no entry in Section 3's table. The Employee record must not assume Firebase Authentication is a permanent choice; it carries only a neutral access-link reference (`userId`), never vendor-specific authentication fields, so a future identity provider change doesn't require redesigning the Employee entity itself.
- **`fieldops_technicians` remains a separate, specialized, unchanged business object** — a field-service resource record, not the enterprise Employee model. Only some Employees are Technicians; this section does not propose renaming, merging, or migrating `fieldops_technicians` into `employees`.

Identity authority, and how an Assignment resolves through it:

```
Employee
  │
  └── linkedUserId ──▶ User
                          │
                          └── Firebase Authentication (credential authority)

Assignment
  │
  ├── resolves ──▶ Employee (workforce identity)
  │
  └── resolves ──▶ User ──▶ Authorization (`firestore.rules` enforcement)
```

**Employee conceptual fields** (proposed/reserved — none of this is implemented yet):

| Field | Status | Notes |
|---|---|---|
| `employeeId` | Proposed | Technical, non-human-readable, immutable document ID. Never a display name or employee number. |
| `employeeNumber` | Future | Human-readable identifier (e.g. `EMP-000143`), distinct from `employeeId`. Not proposed for the initial shape. |
| `displayName` | Proposed | Human-facing name, used by assignment pickers and display snapshots — not an identity key. |
| `firstName` / `lastName` | Proposed | Structured name components, if/when the UI needs them separately from `displayName`. |
| `employmentStatus` | Reserved | `ACTIVE` / `ON_LEAVE` / `INACTIVE` / `TERMINATED` / `RETIRED` / `CONTRACTOR`. The durable business-status model — more expressive than a boolean. An `active` boolean may still exist later as a convenience/query projection derived from this, but `employmentStatus` is the entity's real lifecycle state. Not implemented in code in this phase. |
| `operationalRoles[]` | Proposed | E.g. `PARTS_MANAGER`, `PARTS_ASSOCIATE`, `TECHNICIAN`, `WAREHOUSE_MANAGER`, `WAREHOUSE_ASSOCIATE`, `SERVICE_MANAGER`, `SALES_MANAGER`, `SALES_ASSOCIATE`. Determines assignment eligibility only — never a substitute for the existing security `role` on `users/{uid}`. |
| `companyId` | Future | Company (Section 2) is a Future entity with no implementation today; this field reserves multi-company readiness without implementing organizational structure now. |
| `departmentId` | Future | No Department entity exists yet. Reserved for future org-structure compatibility; not proposed as implemented. |
| `locationId` | Future | Reserved the same way as `companyId`/`departmentId` — no implementation proposed now. |
| `userId` (or `linkedUserId`) | Proposed | Nullable link to `users/{uid}`. Null means the Employee has no application access. |
| `createdAt` / `updatedAt` | Proposed | Standard record timestamps. |

**Operational-role history (reserved, not in scope).** A future, separate business object such as `employee_role_assignments` may become necessary if operational-role *history* (not just current state) is ever required — this is named here for model coherence only, the same "documented concept, not scoped" convention this document already uses elsewhere (see Section 2). No such collection is created by this section, and the detailed schema for it is deliberately not designed here.

**Legacy assignment compatibility.** Workflow records written before this Employee model exists (e.g. Reorder Request's `assignedToUserId`/`assignedBy`/`assignedAt`, Section 4) remain valid as-is and are not rewritten retroactively. See `PROJECT_ARCHITECTURE.md`'s Person Assignment Platform Service Standard for the full display-fallback and legacy-compatibility rules shared by every workflow object, not just Reorder Request.

## 10. Naming recommendation

- **Internal collection/type name: `accounts`.**
- **UI/navigation label: "Customers"** — retained where that's clearer for users, per `ProductBlueprint.md`'s existing navigation.
- `WorkOrder.customerId` (the existing, deployed field on `fieldops_wos`) **may continue pointing to an `accounts` document's ID** — the field is not renamed to `accountId`.
- **Do not rename `customerId` now.** It's a live Cloud Function contract (`functions/src/createWorkOrder.ts`); renaming it is a backend schema change outside this document's scope. The field name and the collection it references are allowed to differ permanently — this is a deliberate, accepted naming mismatch, not a TODO.

It does not update stock yet.") that applying an action to the real ledger needs a trusted Cloud-Function-mediated write path, blocked on enabling Firebase Blaze (a standing platform decision). A "Record Purchase Order" card on `PartDetail.jsx` (assignee-only, shown alongside the existing purchasing-update card while `PURCHASING_IN_PROGRESS`) and a "Reorder Request -- Ordered" card (shown once `ORDERED`, realtime via `hooks/useReorderPurchaseOrders.js`). Removal from the Parts Associate "In Progress" queue happens automatically via existing realtime status filtering. Supplier/Vendor Management, a vendor catalog, a pricing engine, email/vendor communication, receiving, warehouse, bin management, inventory quantity updates, `inventory_actions` changes, `inventory_transactions` changes, Cloud Functions, new routes, and a new Notification Panel section all remain explicitly out of scope.

## 12. Risks and migration notes

- **Scope control**: Company, Equipment/Asset, Vehicle, Invoice, Opportunity/Quote, Service Contract, and PM Schedule are all named for model coherence but are explicitly not built in Sprint 2.0.2 or 2.0.3. Treat any pressure to start building them early as a scope-creep signal, not a sign the model was wrong to include them.
- **`customerId` naming mismatch**: permanent, low-risk, accepted per Section 10 — not something to "fix" opportunistically.
- **No data migration now**: no prior implementation of Account/Contact/Location has ever been written to (the dead `domain/customers.js` store has zero real documents), so Sprint 2.0.2 is a fresh build, not a migration.
- **Avoid creating overlapping people models**: if a general Employee collection is ever built without explicit reference to the existing `users`/`fieldops_technicians` collections, the platform risks a fourth overlapping "person" concept (this codebase has already accumulated multiple parallel "work order" concepts once — see the Design Inventory that preceded this document). Build Employee only when a concrete feature requires it, and always model it as sitting above User/Technician, never replacing either.
