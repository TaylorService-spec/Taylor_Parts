## Sequencing notes

**The `reorder_requests` `allow update` restructuring is a single, indivisible unit -- it lands entirely in PR 3a, not split across 1a and 3a.** The Specification's Technical design shows the Assign transition gaining a new `PARTS_MANAGER`-eligible branch (§ "Restructuring `reorder_requests`' `allow update`") as part of the same restructured block that also carries the four `PARTS_ASSOCIATE` lifecycle branches -- since the entire `allow update` statement is one Firestore Rules block (not eight independent rules), it cannot be partially restructured in 1a and finished in 3a without an intermediate state where the block is neither the original shared-gate shape nor the final per-branch shape. **PR 1a's own scope is therefore limited to the `reorder_requests` READ-side branches and the helpers** (both of which are safe, independently-landable additive `||` widenings per the Specification's own distinction between `allow read`'s single-boolean safety and `allow update`'s shared-gate risk) -- **PR 1a does not touch `allow update` at all.** The `PARTS_MANAGER` Assign-write branch physically lands in PR 3a's full restructuring, alongside the `PARTS_ASSOCIATE` branches, even though it is a `PARTS_MANAGER` capability. **PR 1b (the `PARTS_MANAGER` UI)'s Assign action is therefore gated on PR 3a's Rules deployment, not PR 1a's** -- this is the one place the three "independent" role tracks are not actually independent, and it must be called out explicitly in PR 1b's own PR description so a reviewer doesn't assume 1a alone unlocks it. Every other part of PR 1b (catalog/health, Parts Manager Queue, assigned-work oversight, Relevant History) depends only on PR 1a.

**`inventory_transactions`'s new read branches are identical for `PARTS_MANAGER` and `WAREHOUSE_MANAGER`** (per the Specification's "Why catalog/health reuses the broad read" -- one Rules condition covers both roles). To avoid two PRs racing to add the same two-line branch to the same file, **this branch is landed once, in PR 1a** (the first of the two Rules tracks to reach review), and PR 2a's own diff excludes it, noting in its PR description that `inventory_transactions` was already granted by PR 1a. **PR 2b (`WAREHOUSE_MANAGER` UI)'s catalog/health section is therefore gated on PR 1a's deployment, not PR 2a's alone** -- PR 2a's remaining scope (`inventory_actions`) is what actually gates PR 2b's Part Activity panel specifically.

**Restated, the real dependency graph (not the simplified "three independent tracks" framing the Specification's Executive Summary uses for the general shape):**
- PR 0 -> {PR 1a, PR 2a, PR 3a} (all three Rules PRs need PR 0's fixtures to verify against a real session, though the Rules helpers themselves have no code dependency).
- PR 1a -> PR 1b (catalog/health, Parts Manager Queue, assigned-work oversight, Relevant History only).
- PR 1a -> PR 2b (catalog/health specifically, since `inventory_transactions`'s branch lands in 1a).
- PR 2a -> PR 2b (Part Activity panel specifically).
- PR 3a -> PR 1b (Assign action specifically, since the Assign `PARTS_MANAGER` branch lands in 3a's restructuring).
- **[Correction, recorded post-merge] Issue #100 `employees`-candidate-read Rules PR -> PR 1b (Assign picker READ specifically).** This plan and the Specification both recorded "no `employees` Rules change" for the Assign picker, assuming the `PARTS_MANAGER` could already read its candidate Employee docs. It could not -- the `employees/{employeeId}` block allows only `isAdminOrDispatcher()` or self-read. A separate, security-reviewed Rules PR adds a least-privilege `employees` read branch scoped to the assignment-candidate contract (`ACTIVE` + `PARTS_ASSOCIATE`-eligible + linked user; the exact `buildAssignableEmployeesQuery()` constraints, no new index). **PR 1b is therefore gated on THREE Rules PRs, not two: PR 1a (non-Assign sections), PR 3a (Assign write), and this `employees`-candidate-read PR (Assign picker read)** -- each merged, deployed, and confirmed live before PR 1b.
- PR 3a -> PR 3b (the entire `PARTS_ASSOCIATE` surface).

**Recommended landing order, given the above:** PR 0, then PR 1a (unblocks both 1b's non-Assign sections and 2b's catalog/health), then PR 2a and PR 3a in parallel (independent of each other and of 1a's remaining scope), then 1b/2b/3b once their respective full dependency sets are satisfied -- 1b last among the UI PRs, since it is the only one gated on two separate Rules PRs (1a and 3a).

**Within each Rules PR:** helpers (1a only, since they're shared) and that PR's own new branches land together, verified via SDK/driver-level probes before that PR's own merge -- not split into a "helpers first" sub-PR, since the helpers have no meaning or testable behavior independent of at least one consuming branch.

**Within each UI PR:** the route/nav wiring and the surface's own component tree land together -- there is no safe intermediate state where the route exists but renders nothing, or the component exists but has no route.

## Deployment and rollback boundaries

Restated from the Specification's own "Rollback strategy," expanded per-PR for this Plan's actual sequencing -- consult before authorizing any rollback at any stage of this initiative, present or future.

**Before PR 0 is merged.** No live impact. Normal revert, no ordering constraint.

**After PR 0 is merged, before any Rules PR merges.** `employmentStatus` exists in `AuthContext` state and the nav-gating extension exists in code, but no new route/nav item is reachable (no `operationalRoleAccess`-bearing item exists yet, since none of 1b/2b/3b has landed) and no Rules grant exists yet. Reverting PR 0 at this point is a normal code revert with zero production-data or Rules consequence.

**After a Rules PR (1a/2a/3a) is merged, before it is deployed.** The Rules change exists on `main` but is not live in production -- no behavior change occurs until the separate Deployment Authorization is granted and the deploy actually runs. Reverting the merged-but-undeployed Rules PR is a normal code revert.

**After a Rules PR is deployed and confirmed live, before its UI PR merges.** The new grant is live in production but nothing queries it yet (no UI surface exists to exercise it) -- fully safe to leave live indefinitely even if the corresponding UI PR is delayed, exactly matching this initiative's own precedent (Issue #154's C0 index, left live pending PR C). **Reverting a live-but-unused Rules grant requires its own separate Owner Deployment Authorization to redeploy the reverted rules file** -- merging a revert PR to `main` does not, by itself, undo a live production deployment, the same "merge is not deploy" principle applying in reverse.

**After a UI PR merges.** Reverting a UI PR's frontend code is independent of its Rules grant -- the Rules grant is never automatically removed by reverting the code that used it, mirroring the C0/PR C precedent exactly. **Never revert a Rules PR while its UI PR remains live in production** -- doing so would strand the UI surface with a permission-denied state for every read/write it depends on, recreating exactly the nav-visible-but-permission-denied state Architecture decision 9.4 rejects, except now as a regression rather than an initial-rollout risk.

**PR 3a specifically -- the `reorder_requests` `allow update` restructuring.** If a regression is ever discovered in the live, deployed, restructured rule (e.g., an `admin`/`dispatcher` write that unexpectedly fails, or a `PARTS_ASSOCIATE` write that unexpectedly succeeds on a transition it shouldn't), the safest rollback is redeploying the pre-restructuring rules file in full (not a partial patch), under its own emergency Owner Deployment Authorization, since the restructuring touches every branch's structure simultaneously -- a partial rollback risks leaving the block in neither the original nor the corrected shape.
