## Non-negotiable rules (repeat across every sprint/epic prompt so far)

1. `JOB_STATUS` (`open/assigned/in_progress/complete`) is defined once, in `domain/constants.js`. Never duplicate it. Only `domain/jobActions.js`'s `assignJob()`/`updateJobStatus()`/`createJob()`/`createTechnician()` may write Job/Technician state — no UI component writes directly.
2. **Work Order Engine (Epic 1, merged to `main`) is a separate, real, persisted system**: `fieldops_wos` collection, its own 11-value `WorkOrderStatus` (`CREATED`/`READY_TO_DISPATCH`/`SCHEDULED`/`DISPATCHED`/`ACCEPTED`/`EN_ROUTE`/`ARRIVED`/`WORK_IN_PROGRESS`/`COMPLETED`/`CLOSED`/`CANCELLED`), written ONLY by the `createWorkOrder`/`transitionWorkOrder` Cloud Functions. `firestore.rules` denies ALL direct client writes to `fieldops_wos`/`counters`, unconditionally, no admin exception. See `docs/architecture/ADR-002-work-order-engine.md`.
3. **Two sanctioned write pipelines, never a third**: (1) Job/Technician: component → `domain/jobActions.js` → Firestore directly, rules-enforced. (2) Work Order: component → an actions component (`WorkOrderActions.jsx` for dispatchers, `TechnicianWorkOrderActions.jsx` for technicians, `ExecutionCapture.jsx` for execution data) → `services/workOrderService.ts` → Cloud Function. No page component may ever call `httpsCallable`/`getFunctions` directly, and no page component may write `fieldops_wos`/`counters` directly. There is no `services/workOrderActions.ts` on `main` — that name was carried over from a prior session's stated intent, not a real file; don't cite it. (Same goes for `UI_ACTION_PIPELINES.md`, which also does not exist in this repo.)
4. **Inventory (Epic 2D/3, merged to `main`) is ledger-based, append-only, backend-only** — no mutable "current stock" document anywhere. `inventory_transactions` (RESERVED/RELEASED/CONSUMED) is the sole source of truth for stock movement; `data/partsCatalog.ts` (both client and server mirrors) is metadata-only, no stock authority. See `docs/architecture/ADR-003-inventory-trigger-system.md`.
5. Control Tower (`modules/controlTower/`) is a read-only derived/intelligence layer — must never mutate Firestore directly. (A "Dispatcher Workspace" module was previously referenced here as also existing on `main`; it doesn't -- that work is a "Dispatcher Operations Workspace" on the still-unmerged `epic-2-work-order-interactive-ui` branch. Re-check when/if that branch merges, don't assume it's live.)
6. No second Control Tower implementation, no parallel dispatch logic, no competing domain models, no second Cloud Function competing with `createWorkOrder`/`transitionWorkOrder`, no second inventory system competing with the ledger.
7. **Epic 4 (Warehouse + Fulfillment) adds a physical-reality layer, not a new source of truth**: it must never mutate `inventory_transactions` or `fieldops_wos`, must never introduce an event/pub-sub system, and reconciliation is strictly read-only reporting (no auto-correction). See "Epic 4" section below.
8. **The Operations dashboard (`modules/operations/Operations.jsx`) is a READ-ONLY EXECUTIVE/MONITORING layer, explicitly NOT a second dispatcher tool.** `modules/dispatch/Dispatch.jsx` (a domain-routing scaffold briefly moved this to `domains/execution/ExecutionWorkspace.jsx` on PR #22, then reverted back to its original location/name during that same PR's cleanup pass -- it's `modules/dispatch/Dispatch.jsx` again, don't assume otherwise without checking) remains the only place a human assigns a job to a technician; Control Tower remains the dispatcher's real-time operational intelligence view. Operations must never grow an "assign"/"dispatch"/"act on this job" affordance — its scope is inventory/warehouse/procurement reporting only. This rule exists specifically to prevent "two dispatchers + one dashboard + one control tower" product fragmentation, flagged explicitly when PR #21 was approved for merge.
9. **Work Order Lifecycle Authority**: the canonical lifecycle is defined by `functions/src/transitionEngine.ts` (mirrored client-side in `domain/workOrderWorkflow.js`). All UI, docs, tests, and future enhancements must conform to this implementation -- if a doc (including this one) and the code differ, the transition engine wins until an intentional, versioned change is approved. See `docs/architecture/ADR-002-work-order-engine.md`'s "Work Order Lifecycle Authority" section for why this was added (a pasted spec described a plausible but factually wrong transition table for Epic 2 Phase 2B; caught via a live `getAllowedActions()` smoke test before any code changed).
10. **Work Order execution data (`qtyUsed`/`executionLog`/`lastUpdated`) is written EXCLUSIVELY via `functions/src/updateWorkOrderExecutionData.ts` (Epic 6 Phase 6.3), Admin SDK only.** `firestore.rules`' `fieldops_wos` rule must NOT be loosened to allow these fields from the client -- Admin SDK bypasses rules entirely, so no client-side exception is needed or should ever be added. This function never touches `status`/`assignedTechId`/any lifecycle timestamp field; it is entirely separate from `transitionWorkOrder()`/`transitionEngine.ts`, both structurally and in what it's permitted to write.
11. **Epic 7 is finalized as the Execution Intelligence Layer.** `field-ops-app-vite/src/analytics/executionAnalyticsService.ts` (PR #34, merged) is a frozen analytics contract — `getTechnicianExecutionStats()`, `normalizeQtyUsed()`, and `getWorkOrderExecutionSummary()` must not be redefined, forked, or reimplemented by downstream systems. Epic 8+ cross-domain work (`analytics/operationsIntelligenceService.ts`) must import and compose these functions, never duplicate their logic.
12. **Operational intelligence layers (Epic 8+, `analytics/operationsIntelligenceService.ts` and anything downstream of it) must remain strictly compositional** — aggregation and cross-domain correlation only, never a second computation surface. Any metric requiring arithmetic or normalization over execution data (technician stats, work-order-execution summaries, `qtyUsed` normalization) must originate from `executionAnalyticsService.ts` (rule 11); any metric requiring arithmetic over inventory forecasting/risk (usage rate, reorder point, stockout risk) must originate from `domain/inventoryAnalyticsEngine.ts` (Epic 3). A downstream layer may filter, count, join, or correlate those outputs by key, but must not reimplement the math itself.
13. **React Router (`react-router-dom`) is now intentional and accepted, as of Sprint 2.0.1 (Release 2.0, PR #41).** This reverses rule 8's/PR #22's prior state, deliberately: PR #22 tore out an earlier routing scaffold as a scope-convergence decision ("structural only, not wired in," per that PR's own body), not a permanent ban — audited before Sprint 2.0.1 reintroduced it, confirming no technical/security/deployment/GitHub-Pages blocker was ever the actual reason. `field-ops-app-vite/src/navigation/navConfig.js` is now the navigation source of truth (business-domain/sub-nav tree), consumed by real routes in `App.jsx`/`navigation/AppShell.jsx` — not `App.jsx`'s old flat `NAV` array, which no longer exists. See `docs/Architecture.md`'s "SPA routing" section and `docs/architecture/SYSTEM_AUTHORITIES.md`'s "Navigation" row.
14. **Employee Identity and Platform Assignment Standard (governance-approved, not yet implemented — see `docs/PROJECT_ARCHITECTURE.md`'s "Person Assignment Platform Service Standard" and `docs/BusinessEntityModel.md` Section 8a).** Employee (`employees/{employeeId}`) is the authoritative workforce identity; User (`users/{uid}`) is the authoritative application-access identity; Firebase Authentication is the credential authority only — the three are never merged into one entity. Any feature requiring a person assignment must reuse this one Platform Assignment Standard (`assignedToEmployeeId`/`assignedToUserId`/`assignedToDisplayName`/`assignedByEmployeeId`/`assignedByUserId`/`assignedAt`) — do not introduce manual Firebase UID inputs or feature-specific user-directory logic in any domain. Operational roles (e.g. `PARTS_ASSOCIATE`, `TECHNICIAN`) determine assignment eligibility; security roles (`admin`/`dispatcher`/`technician`) continue to determine application permissions — these remain separate role concepts. `users/{userId}`'s existing self-read/no-client-write `firestore.rules` posture is not weakened by this standard.

## Sprint 2.0.4 direction: three false starts, then a real answer

Worth reading before anyone picks this back up — "Sprint 2.0.4" was reframed three times in one session, and the reasoning for rejecting the first two matters:

1. **First framing: "Cloud Functions Deployment Readiness."** A full implementation plan was written and approved, emulator validation was run for real (all 3 Cloud Functions exercised directly, catching and fixing a genuine bug — see PR #49), and the plan was ready to request production-deploy authorization. **Then the user clarified: not using Blaze right now, full stop.** This wasn't a "not yet" — it was an explicit standing decision. The emulator work and the bug fix it found were real and worth keeping (recovered into PR #49); the deployment itself never happened and is not imminent.
2. **Second framing: "Spark-Compatible Work Order Enablement."** A full implementation plan was drafted to rebuild Work Order creation/transitions as client-direct Firestore writes, avoiding Cloud Functions entirely. **This was explicitly rejected before any code was written** — the user's long-term vision includes paid customer deployments, customer-owned Snowflake, ETL/BI/ERP integrations, none of which should be designed around a temporary Spark-plan constraint. Rebuilding the write path around Spark's limits risked becoming the permanent architecture through sheer inertia. No code from this framing was ever built.
3. **Third framing, the one that stuck: "Deployment Mode Strategy."** Rather than pick one write-path architecture, define **deployment modes** as a first-class, permanent concept (Development / Demo / Managed Hosted / Enterprise Integration) so the platform can explicitly know which mode it's running in, rather than one mode being silently assumed forever. Now written and committed as [`DeploymentModeStrategy.md`](DeploymentModeStrategy.md) — see that document for the full definition (modes, tenant strategy, configuration strategy, data ownership, integration expectations).

**Current, accurate state**: `functions/src/*.ts` (`createWorkOrder`/`transitionWorkOrder`/`updateWorkOrderExecutionData`) are correct and bug-fixed (PR #49) but still undeployed, by deliberate choice, not blockage. Sprint 2.0.3's UI is complete and shouldn't need rebuilding. `DeploymentModeStrategy.md`, `PlatformOperatingModel.md`, and `IntegrationArchitecture.md` are all now written — the governance documentation roadmap is complete. No named integration (ERP, BI/Snowflake, accounting, CRM, AI) is implemented or scheduled; `IntegrationArchitecture.md` defines the architecture any future one would need to conform to, not a build plan.

## Standing operating rule: verify, don't assume

Established after an incident where a prior turn assumed a PR had merged because it had been *discussed* as the next step, when it hadn't actually happened yet (PR #12 is exactly that pattern, still unresolved) — and reinforced again when a prior session's context doc cited four architecture docs by path (`ADR-001`, `UI_ACTION_PIPELINES.md`, `EPIC-2.md`, and originally `ADR-003` before it was actually written) that didn't exist in the repo. Before recommending merge order, rebase necessity, deploy state, "what's next," or citing any doc by path: run `git fetch`, `gh pr view --json state,mergedAt,mergeable,mergeStateStatus`, `git log origin/main`, and `Glob`/`Grep` for the doc path first. Conversation history or a prior session's notes describing something as existing is not evidence it does. Same for Firestore rules/Cloud Functions deploy state — committed/merged is not deployed; query the live project directly, don't trust a doc's snapshot or a console screenshot.

Also verified this session: a chained shell command (`git add && git commit && git push origin main`) that gets denied by a permission check is denied as a whole — none of it runs, including the `add`/`commit` portion, even though only the final `push` was the actual concern. Don't assume a leading step in a blocked chained command executed; check `git log`/`git status` after any denial before building on top of it.

**The single strongest concrete case for this rule, found 2026-07-09**: the live `taylor-parts` project's deployed `firestore.rules` had never included a `reorder_requests` match block at all, for the entire span of Sprints 2.1.3 through 2.1.7 — every PR in that span passed `npm run build`/lint/code review and was merged as if complete, but the actual write path had been failing on `permission-denied` in production the whole time, invisibly, because nothing in this project's process ever re-queried the live deployed ruleset after merging a rules change. Found only by investigating a user-reported "Request Reorder does nothing" bug, via a read-only Admin SDK check (`getSecurityRules().getFirestoreRuleset()`). Fixed by an explicit `firebase deploy --only firestore:rules`, with the user's authorization, and re-verified live afterward. **The general lesson: after any `firestore.rules` change merges, the rules still need their own explicit deploy step — a merged PR is not that step, and nothing automates it.**

- **GitHub Actions `pull_request`-triggered checks occasionally get stuck `queued` with no runner ever assigned** (`runner_id: null` for many minutes), while the equivalent `push`-triggered build for the identical commit completes normally in seconds. ## `admin-check` — a self-authored, rules-bypassing diagnostic tool (not a Cloud Function)

`.claude/skills/admin-check/` (`lib.js` + `SKILL.md`, committed on the stale, unmerged `feature-dispatch-control-tower` branch — files persist in the local working tree across branch switches even though not on `main`) is a real, self-authored `firebase-admin` helper for privileged, rules-bypassing reads/writes against the live `taylor-parts` project when a ground-truth answer is needed that the client SDK/rules can't give (e.g. "what does the live deployed ruleset actually say," "does this document really exist"). It caches a service-account key's *file path* only (`keypath.local.txt`, gitignored) — never the key itself. This is scratch/debugging tooling, not application code — never wire it into `field-ops-app-vite`, never add `firebase-admin` to that package's dependencies. Using it for anything beyond a read-only ground-truth check (especially any write) needs the user's explicit, specific authorization each time — it is exactly the kind of "credential-bypassing helper performing a production write" pattern this environment's safety layer is designed to stop, and repeated confirmations don't substitute for a clear, specific ask.
